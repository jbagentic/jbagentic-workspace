## slide-07
Why Not Just Prompt It?
Prompting teaches behavior. MCP defines capability.
Prompt Only
Re-explain rules repeatedly
Higher chance of inconsistent behavior
More correction tokens
Session compression may lose context
Model may invent the process
Prompts guide behavior...
...but outcomes are probabilistic.
VS
MCP Tool Contract
Named tools
Required fields
Valid categories
Validation rules
Predictable outputs
Reusable capability boundary
Structured tool calls with validation.
Example tools
add_expense
expense_summary
calculate
Expense recording needs correctness.
The agent should not freely invent records or generate SQL directly.
Syntax can look correct while the actual inserted data is wrong.
MCP can save tokens because less time is spent correcting behavior and reteaching rules.
If the session is cleared or compressed, prompts may be forgotten, but tool contracts stay more stable.
MCP reduces guesswork by giving the agent a well-defined tool boundary.

list_categories prevents the agent from inventing categories.
calculate prevents the model from doing money arithmetic by itself.
Bulk tools handle multiple expenses in one transaction.

For fuzzy input, use AI. For money, validation, and calculation, use tools.

Problem Type | Best Handled By | Example
Category validation | Tool / Schema (MCP) | Fixed category list (食物 / 交通 / ...)
Tax and calculation | Deterministic tool | calculate ("10.5 + 3.5")
Data validation & rules | Tool / Schema (MCP) | Required fields, format, ranges

SQL syntax may be correct, but interpretation may be wrong.
Leads to wrong data and repeated corrections.
Wastes tokens and increases cost.

## slide-11
MCP or CLI?
A discussion with my mentor Hui Ming, and what I learned.
Context: I had a discussion with my mentor Hui Ming: should agents use CLI tools or MCP?
Balanced Takeaway
CLI is powerful, familiar, and has existed for a long time.
LLMs already understand many common CLI patterns.
CLI can work well when the command is simple and the output is easy to inspect.
But CLI still usually needs a skill, prompt, or instruction layer to teach the agent how to use it, what parameters to pass, what output means, and what errors look like.
MCP comes with a tool schema and structured contract out of the box.
CLI
Command Line Interface
CLI works well when...
Command is simple
Output is easy to read
Tool is familiar
Human can inspect result
A skill / prompt can teach usage
Fast exploration is needed
MCP
Model Context Protocol
MCP works well when...
Input needs structure
Validation matters
Repeatability matters
Agent must act predictably
Tool schema should describe usage
Workflow should be safer
How much guesswork do we want to leave to the AI?
Both have value. The difference is the amount of structure you want.
CLI
More freedom
More guesswork
Choose based on the problem
MCP
More structure
Less guesswork
I still plan to experiment with custom CLI tooling. For this expense agent, MCP gave me the control and contract I wanted.

## slide-12
Where Things Went Wrong
Real AI apps fail at boundaries. Improving the system means designing better boundaries.
1 Voice STT
Problem
- Chinese speech can be mis-transcribed.
- Example: "六合仙" may become "六八仙".
Fix direction
- Better phrasing from user
- Better STT model
- Clarification flow when confidence is low
六合仙 (wrong)
六八仙 (correct)
2 Receipt Photo Context
Problem
- Receipt image may be interpreted as related to a previous record.
Fix direction
- Clearer prompt / tool flow
- Attachment intent handling
- Ask for confirmation if context is uncertain
Confirm intent before linking
3 Tax & Calculation
Problem
- LLM may calculate tax or multiplication wrongly.
- "消费税 六" may be interpreted as RM6 instead of 6%.
Fix direction
- Use calculate tool for math
- Memory / rules for tax logic
- Explicit percentage handling (6% ≠ RM6)
消费税 六 → 6% (correct)
RM6 (wrong)
Study failures. Improve boundaries. Build a more reliable agent.
