## slide-04
Two different AI era: Browser vs. Code Editor

Browser AI (The Outside Consultant)
- Repeat copying, pasting, and switching windows to get solutions.
- AI has Zero Context: Requires manually describing imports, types, helpers, and more.

Editor-Native AI (The Resident Partner)
- With this, NO more copy & paste & switching browser, AI lives directly in your workspace to read/write files.
- Deep Awareness: Background indexing of files, utilities, and styling standards, so you don't need to repeat what you using, what's the product logic every time.

## slide-14
Case study 3: Superpower plugin

- Whenever our team trying to do a new feature, normally we will use this `superpower plugin`.
- `Superpower` is a software development methodology for coding agent, its a public repo on github
- How it work:
  - When you trigger this superpower, first he will digest your requirement (from chat), then they will give you a proposal, or we can say its a coding plan with details like what files he will be creating (according to your coding standard rules), come up with core logic functions with test cases in a .md file.
  - After you reviewed the .md file, you can chat and make any changes you want, then you sign off (to give green light to agent & proceed).
  - Finally, Claude Agent will follow the .md design file to start coding & designing, then Claude will create parallel agents to test the written code, fix bug & test & repeat until all good.

## slide-18
Model Context Protocol (MCP)

MCP is a standardized interface for AI agent, we can use MCP to connects to various of data sources and tools, enabling the AI to retrieve or post data easily.

The Old Way (Custom Cable Mess)
- API Chaos: Writing custom API code for every external connection.
- Fragile: Integrations easily break whenever the tool API structures change.

The MCP Way (The Universal USB)
- One Port: Exposes tool capability natively directly inside Claude.
- Plug-and-Play: Zero custom integration glue code required. Easy setup thru jsonb file

## slide-19
My next move: Connecting Claude to Sentry (error logging applications)

Connecting Sentry to our Claude allows it to instantly access production stack traces and error contexts without me having to manually copy-paste logs. This enables the agent to provide automated, accurate diagnoses and suggest precise fixes directly within the development environment

Phase | Before (The 'Hunting' Phase) | After (The 'Guided' Phase)

Search | Hunting through raw metrics and diagnostic dashboards (30m+). | Clicking the alert link; Claude immediately maps and opens local file.

Guess | Attempting to author a fix matching current patterns manually. | Claude suggests an isolated, standards-compliant patch matching CLAUDE.md.

Verify | Locally running tests inside terminal suites manually. | Claude invokes local test commands to validate stability before commit.

## slide-20
Why We Moved: Cursor vs. Claude

Understand that Cursor & Claude actually behave quite similar, but in the end we still choosing claude for our teams is because of we are product based company, so our product logic are customized and heavy, so in the end we choose quality over speed.

Metric | Cursor (The Typist IDE) | Claude (The System Architect)

Active Chat Context | Chat Pollution: Fills chat window with code blocks, leading to slow replies. Whenever you doing a big feature, the response will not be too accurate | Context Isolation: Spawns parallel background subagents to work behind the scenes & give answer only to the origin chat agent.

Execution Logic | Inline focus: Predicts autocomplete steps dynamically while typing. (Very good for inline logic & redundant codes) | Slower, no auto-complete suggestion

Task Size & Speed | Fast Response: Near-instant response. Perfect for small prompt sizes, quick debugging, and micro-queries. | Slower but Smarter: Takes longer to process, but the resulting logic is significantly more accurate.
